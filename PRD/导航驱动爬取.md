# 导航驱动爬取（Navigation-Driven Crawl）

> 状态：第一阶段已完成（`nav` 子命令已交付）；第二阶段规划中（简化方案：直接抓取菜单 URL 对应页面，不做递归链接抓取）
> 作者：webextract 维护者
> 创建日期：2026-06-28

---

## 1. 背景：现有 crawl 为什么「抓取不全」

当前 `webextract crawl` 采用**纯 BFS**：从入口页出发，对每一层页面（含列表页、详情页）都执行同一套逻辑——

```
抓取页面 → 提取正文 Markdown → 从整页 <a href> 提取所有站内链接 → 入队
```

这套逻辑在「文档站」场景工作良好（文档结构扁平、每页都是「正文 + 左侧导航」），但在**内容站/门户站**（新闻、资讯、电商等）下暴露出结构性问题：

| 症状 | 根因 |
| --- | --- |
| 抓取偏离主题，大量「相关阅读」「热门排行」页面涌入 | 详情页里的非正文链接被无差别提取并入队 |
| 列表页/详情页深度混淆，抓不全分类 | BFS 按距离种子页的跳数分深度，与真实「分类→列表→详情」的层级无关 |
| 同一分类的多页内容漏抓 | 受 `--depth` 限制，列表的「下一页」翻页链往往在第 2~3 层就被截断 |
| 输出无分类归属 | 文件按 URL 路径落盘，不体现站点的导航分类结构 |

**核心问题**：BFS 把**结构性的三层（导航 / 列表 / 详情）**坍缩成了**单一的「距离」维度**，并用「提取所有链接」的暴力策略替代了「按角色提取不同链接」的分层策略。

## 2. 目标

用**导航驱动 + 角色分层**的三层架构替代纯 BFS：

```
第一层 骨架（广度，仅一次）        第二层 列表流（深度，线性翻页）    第三层 详情页（纯抽取）
首页 → 提取导航 → 分类队列  →→→  分类列表页 → 详情链接队列        →→→  仅解析正文/图片/时间
                              ↘  翻页链 → 下一页（线性走完）↗
                              ❌ 详情页绝不提取任何链接（防被带偏）
```

- **第一层**：从首页提取**一次**导航栏链接，得到分类种子（如 新闻 / 体育 / 科技）。
- **第二层**：逐个消费分类，访问列表页 → 提取**当前页所有详情页链接**入详情队列 → 顺「下一页」线性翻页直到走完。翻页是线性的（类深度），不再广度扩散。
- **第三层**：批量消费详情队列，**只解析正文、图片、时间等核心数据**，**绝不**在详情页提取任何链接——从源头杜绝「相关阅读」「热门排行」的污染。

## 3. 非目标（本期不做）

- 自动登录 / 鉴权抓取。
- 通用反爬对抗（验证码、JS 挑战）——仍依赖现有 chromedp 渲染能力。
- 详情页正文之外的结构化字段（如价格、SKU）的精细 schema 抽取（仅做正文 + 图片 + 时间）。

---

## 4. 整体设计与分阶段

整个能力拆为**两个阶段**，对应两个独立子命令，互不耦合：

| 阶段 | 子命令 | 交付物 | 状态 |
| --- | --- | --- | --- |
| 第一阶段 | `webextract nav <URL>` | 从首页提取导航菜单树（菜单名 + URL），输出树形 / JSON | **本期开发** |
| 第二阶段 | `webextract sitemap-crawl`（暂定名） | 基于导航种子的三层抓取，按菜单分类组织输出 | 后续 |

设计上，第一阶段的 `nav` 输出的 **JSON** 即第二阶段的**输入**（可管道传入或读文件）。这样两个阶段可以独立调试：先用 `nav` 看清站点导航结构，确认无误后再驱动抓取。

## 5. 第一阶段需求：`webextract nav` 命令

### 5.1 功能定义

输入一个网站 URL，渲染首页，**提取主导航菜单结构**，输出包含「菜单名称 + 链接 URL」的树形结构。

### 5.2 命令行接口

```text
webextract nav <URL> [选项]
```

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `--format` | `tree` | 输出格式：`tree`（人类可读树形）/ `json`（机器可读，供第二阶段消费） |
| `--max-depth` | `0` | 导航菜单最大层级（1=仅一级，2=含二级子菜单，0=不限） |

### 5.3 数据结构（NavItem 树）

```go
type NavItem struct {
    Text     string    `json:"text"`               // 菜单文本（已去空白）
    URL      string    `json:"url,omitempty"`       // 规范化后的绝对 URL（无链接时为空，如分组标题）
    Children []NavItem `json:"children,omitempty"`  // 子菜单
}
```

### 5.4 Tree 输出示例

```text
网站导航 — https://example.com

├─ 新闻 /news
│  ├─ 国内 /news/domestic
│  └─ 国际 /news/world
├─ 科技 /tech
│  └─ 人工智能 /tech/ai
├─ 体育 /sports
└─ 关于我们 /about （站外，已过滤）
```

### 5.5 JSON 输出示例（第二阶段输入）

```json
{
  "seed": "https://example.com",
  "items": [
    {
      "text": "新闻",
      "url": "https://example.com/news",
      "children": [
        { "text": "国内", "url": "https://example.com/news/domestic" },
        { "text": "国际", "url": "https://example.com/news/world" }
      ]
    },
    { "text": "科技", "url": "https://example.com/tech" }
  ]
}
```

### 5.6 导航提取算法（关键技术点）
需要考虑页面导航的复杂性，尽量贴近用户肉眼可见的导航栏结构。

### 5.7 验收标准（第一阶段）

- [x] `webextract nav <真实站点>` 能输出可读的树形菜单，含菜单名与链接。
- [x] `--format json` 产出结构合法、可被第二阶段解析的 JSON。
- [x] 含二级子菜单（下拉）的站点能正确输出嵌套结构。
- [x] 导航中的「分组标题（无链接）」能作为父节点正确归类子项。
- [x] 重复链接与站外社交链接按规则被过滤/标注。
- [x] `go build` 通过，`go vet` 无告警。

## 6. 第二阶段需求（规划，本期不实现）

### 6.1 功能定义

基于第一阶段提取的导航菜单 URL，**直接抓取每个菜单项对应网页的内容**（正文 Markdown），不做下一层链接的递归抓取。输出严格按照菜单的树形结构组织文件夹。

**核心简化**：不执行三层调度（导航→列表→详情），而是将导航菜单中的每个 URL 视为一个独立抓取目标，仅抓取该 URL 页面本身的内容，不提取页面中的任何链接进行后续抓取。

### 6.2 命令行接口（草案）

```bash
# 方式 A：现场提取导航再抓取（一步到位）
webextract sitemap-crawl <URL>

# 方式 B：复用第一阶段产出的 nav.json（可人工编辑/裁剪后再抓）
webextract sitemap-crawl --nav nav.json
```

新增参数：

| 参数 | 说明 |
| --- | --- |
| `--nav` | 读取第一阶段 JSON 作为抓取目标（与 `<URL>` 二选一） |
| `--nav-depth` | 使用导航的前 N 层菜单作为抓取目标（默认全部） |
| `--max-depth` | 导航菜单最大层级（1=仅一级，2=含二级子菜单，0=不限），与第一阶段 `nav` 命令一致 |

### 6.3 抓取逻辑

1. **获取菜单 URL**：
   - 方式 A（`<URL>`）：内部调用第一阶段 `ExtractNav` 提取导航菜单树，得到所有菜单项的 URL。
   - 方式 B（`--nav`）：读取第一阶段产出的 JSON 文件，解析为 `NavItem` 树。
2. **逐页抓取**：遍历菜单树中的每个带 URL 的节点，**仅抓取该 URL 对应页面的内容**（复用现有 `extractFromHTML` 提取正文 Markdown），**不提取页面中的任何链接**进行后续抓取。
3. **无链接节点跳过**：菜单中「分组标题（无 URL）」的节点仅用于构建目录层级，不产生抓取任务。
4. **URL 去重**：如果多个菜单项指向同一 URL，仅抓取一次，但在对应的所有菜单路径下生成文件（或使用软链接/符号引用）。

### 6.4 去重与边界控制

- **URL 去重**：使用 `visited` 集合（规范化 URL 为 key），同一 URL 仅抓取一次。
- **熔断/限流**：复用现有 worker pool + rate limiter + 连续失败熔断。

### 6.5 输出组织

输出文件夹结构**严格按照菜单的树形结构**生成：

```
output/
├── index.json        # 索引：每页带 菜单路径/URL
├── index.md          # 按菜单分类分组的人类可读索引
├── 新闻/
│   ├── 国内.md       # 对应菜单「新闻 > 国内」的页面内容
│   └── 国际.md       # 对应菜单「新闻 > 国际」的页面内容
├── 科技/
│   └── 人工智能.md   # 对应菜单「科技 > 人工智能」的页面内容
└── 体育.md           # 对应菜单「体育」的页面内容
```

**规则**：
- 菜单树的每个层级对应一层目录。
- 带有 URL 的叶子节点（或无子菜单的节点）生成一个 `.md` 文件，文件名为菜单文本。
- 无 URL 但有子菜单的节点（分组标题）仅创建目录，不生成文件。
- 文件落盘**按菜单路径**组织，满足「按菜单栏结构排序」的要求。

## 7. 后续增强（Backlog）

- **hover 子菜单模拟**：对需 hover 才渲染的下拉菜单，用 chromedp `MouseHoverNode` 逐项触发后重读 DOM。需评估成本与命中率。
- **详情页时间/图片抽取**：第三层增加结构化的发布时间、首图抽取（meta og / time 标签）。
- **导航质量评分**：自动评估提取到的导航是否「像主导航」（链接数、与首页 host 一致性、文本特征），过滤误抓的页脚/面包屑。
- **sitemap.xml 兜底**：站点若有 `sitemap.xml`，可作为导航提取的补充/校验源。
